# MVP Core User Story Test Scenarios

Документ описывает базовые E2E/интеграционные тест-сценарии (A–E) для MVP и связывает их с user stories из PRD и приёмочными критериями раздела 10.

## 1) Coverage map: user stories → тесты A–E

| User story из PRD (раздел 4) | Описание покрытия | Тесты |
|---|---|---|
| US-1: Создание сейфа → верификаторы → блоки/получатели → публикация | Полный happy-path создания и базовые валидации | A, B |
| US-2: Событие → инициация → кворум → grace 24ч → финализация и доступ | Событийный pipeline, кворум, таймер, раскрытие | C, D |
| US-3: Heartbeat блокирует раскрытие при активности за 365 дней | Анти-фрод/защитная логика до кворума/финализации | E |

## 2) Детализация сценариев

### Сценарий A — Создание сейфа с 3 верификаторами и кворумом 2 (happy path)

**Regression labels:** `smoke`, `core`, `regression`

**Traceability to PRD AC (п.10):**
- AC-10.1: создание/редактирование сейфа, N=3 верификаторов, кворум 2.

**Preconditions**
- Есть зарегистрированный и подтверждённый по e-mail аккаунт владельца.
- Пользователь авторизован.
- Подготовлены 3 валидных e-mail верификаторов и минимум 1 получатель.

**Steps**
1. Открыть форму создания сейфа.
2. Заполнить обязательные поля сейфа.
3. Добавить 3 уникальных e-mail верификаторов.
4. Установить кворум подтверждений = 2.
5. Добавить минимум один блок с контентом и получателем.
6. Сохранить/опубликовать сейф.

**Expected Results**
- Сейф создан и виден в списке сейфов владельца.
- Для сейфа сохранены ровно 3 верификатора.
- Кворум сохранён как 2.
- Блок(и) и назначенные получатели сохранены.

**Negative Checks**
- Нельзя сохранить сейф с <3 верификаторами.
- Нельзя установить кворум > числа верификаторов.
- Нельзя добавить дублирующий e-mail верификатора.

### Сценарий B — Редактирование сейфа и базовая валидация ограничений

**Regression labels:** `smoke` (basic), `core`, `regression`

**Traceability to PRD AC (п.10):**
- AC-10.1: редактирование сейфа и сохранение параметров верификации.

**Preconditions**
- Существует сейф, созданный по сценарию A.
- Владелец авторизован.

**Steps**
1. Открыть существующий сейф в режиме редактирования.
2. Изменить описание/метаданные сейфа.
3. Обновить один из блоков (контент/получателя).
4. Сохранить изменения.
5. Повторно открыть сейф.

**Expected Results**
- Изменения persist-ятся и отображаются после повторного открытия.
- Конфигурация верификаторов (3) и кворума (2) остаётся валидной.

**Negative Checks**
- Попытка снизить число верификаторов до 2 отклоняется валидатором.
- Попытка установить кворум 3 при ожидаемой политике MVP 2/3 подсвечивается как несоответствие конфигурации smoke-профилю.
- Попытка сохранить пустой обязательный блок отклоняется.

### Сценарий C — Инициация события и достижение кворума 2 из 3

**Regression labels:** `core`, `regression`, `security-lite`

**Traceability to PRD AC (п.10):**
- AC-10.2: инициация события (смерть/недееспособность) и сбор ≥2 подтверждений.

**Preconditions**
- Есть активный сейф с 3 верификаторами и кворумом 2.
- Heartbeat владельца не блокирует событие (не было логина >365 дней или тестово отключён).
- Доступны аккаунты/инвайты верификаторов.

**Steps**
1. Инициировать событие типа «смерть» (или «недееспособность»).
2. Проверить, что статус процесса стал `pending_quorum` (или эквивалент).
3. Подтвердить событие первым верификатором.
4. Подтвердить событие вторым верификатором.

**Expected Results**
- После 2-го подтверждения кворум считается достигнутым.
- Система переводит кейс в стадию ожидания grace-периода (24 часа).
- Аудит-трейл фиксирует инициацию и каждое подтверждение.

**Negative Checks**
- Повторный голос того же верификатора не увеличивает счётчик.
- Голос неавторизованного/непривязанного пользователя отклоняется.
- До достижения кворума финализация недоступна.

### Сценарий D — Grace 24 часа и финализация раскрытия (happy path)

**Regression labels:** `smoke` (happy), `core`, `regression`, `security-lite`

**Traceability to PRD AC (п.10):**
- AC-10.2: таймер 24 ч и финализация после кворума.
- AC-10.4: публичная ссылка 24 ч + поведение после истечения срока.

**Preconditions**
- По сценарию C достигнут кворум 2/3.
- Кейс перешёл в состояние grace-периода.
- В одном из блоков настроена публичная ссылка.

**Steps**
1. Проверить, что до истечения 24 часов финализация не выполнена.
2. Дождаться/эмулировать истечение grace = 24 часа.
3. Запустить (или дождаться авто-) финализацию.
4. Открыть выданную публичную ссылку на блок.
5. Проверить заголовки/атрибуты noindex и наличие CAPTCHA.
6. Дождаться/эмулировать +24 часа жизни ссылки и повторно запросить URL.

**Expected Results**
- Финализация выполняется только после истечения grace.
- Получатели/целевые субъекты получают доступ согласно настройкам блока.
- Публичная ссылка доступна ограниченное время (24 часа).
- По окончании TTL ссылка возвращает `410 Gone`.

**Negative Checks**
- Попытка принудительно финализировать до окончания grace отвергается.
- Прямой доступ к публичному блоку без прохождения CAPTCHA блокируется.
- Повторное использование протухшей ссылки не восстанавливает доступ.

### Сценарий E — Heartbeat блокирует раскрытие

**Regression labels:** `smoke` (basic), `core`, `regression`, `security-lite`

**Traceability to PRD AC (п.10):**
- AC-10.3: логин владельца в последние 365 дней блокирует раскрытие.

**Preconditions**
- Есть активный сейф с конфигурацией 3 верификатора / кворум 2.
- Зафиксирован успешный логин владельца в интервале <365 дней до инициации события.

**Steps**
1. Инициировать событие для сейфа.
2. Проверить решение системы по heartbeat-правилу.
3. (Опционально) попробовать собрать подтверждения верификаторов после блокировки.

**Expected Results**
- Событие автоматически отклоняется (или переводится в статус blocked_by_heartbeat).
- Процесс кворума/финализации не стартует.
- В аудите присутствует причина отклонения: heartbeat within 365 days.

**Negative Checks**
- Нельзя обойти блокировку heartbeat ручной финализацией.
- Голоса верификаторов после heartbeat-блокировки не должны менять итог.
- Нельзя создать публичную ссылку в рамках отклонённого события.
